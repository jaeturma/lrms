# LRMS Module Inventory

Generated by direct inspection of the codebase (routes, controllers, models, migrations, services, form requests, Inertia pages, and feature tests). No application code was modified to produce this document. Completion percentages are an engineering judgment call based on CRUD completeness, validation, authorization, UI, and test coverage observed in code — not a business/product estimate.

Roles referenced below come from `app/Http/Middleware/EnsureUserRole.php` + `routes/web.php` role groups:

- **adminWorkspaceRoles**: `admin, superadmin, sysadmin, ito, manager, librarian, supply, cidchief, asds, sds` — base gate for the whole `/app/admin` workspace.
- **catalogRoles**: `admin, superadmin, sysadmin, ito, manager, librarian, supply` — manage catalogs (titles, equipment catalogs, digital materials).
- **systemAdminRoles**: `admin, superadmin, sysadmin, ito` — reference data, settings, content, distributions, import.
- **school**: single role for school-side accounts (one user per school, linked via `users.school_id`).

Authorization is enforced entirely through route middleware (`role:...`) and manual `abort_if()` ownership checks inside controllers — there are **no Laravel Policy classes** anywhere in the app.

## 1. Authentication & Account Security

**Purpose:** Sign school users and division/admin staff into the correct workspace, and let them recover access.

**Current Completion:** 85%

**Existing Features:**
- School-user login via Laravel Fortify, restricted to `role = 'school'` (`FortifyServiceProvider::authenticateUsing`).
- Separate admin/staff login (`AdminAuthController`) restricted to the 10 admin-workspace roles; auto-logout if a non-admin role slips through.
- OTP-based password reset (`PasswordResetOtpController`): 6-digit code, 10-minute expiry, cached (not persisted), rate-limited (`throttle:6,1`).
- Two-Factor Authentication (TOTP) via Fortify, with `confirm` and `confirmPassword` enabled.
- Login throttling (5/min per email+IP) and 2FA throttling (5/min).
- Session regeneration/invalidation on login/logout.

**Missing Features:**
- Fortify `registration()` and `emailVerification()` features are enabled in config but unused/unreachable in the UI (`canRegister: false` on the login page) — dead configuration.
- No login/security audit log (who logged in, when, from where).
- No account lockout after repeated failures beyond the per-minute rate limiter.
- No "resend OTP" endpoint distinct from resubmitting the whole form.
- No session management UI (view/revoke other active sessions).

**Dependencies:** `App\Models\User`, `App\Actions\Fortify\CreateNewUser`, `App\Services\AppSettingsService` (SMTP must be configured for OTP emails to send).

**Database Tables:** `users`, `password_reset_tokens`, `sessions`, `cache` (OTP storage).

**Routes:** `admin.login`, `admin.login.store`, `admin.logout`, `login` (Fortify), `password.request`, `password.otp.send`, `password.otp.verify`, `password.otp.update`, `two-factor.*` (Fortify).

**Permissions:** Admin login accepts the 10 admin-workspace roles; the default Fortify login accepts only `school`.

## 2. User Account Settings

**Purpose:** Let any authenticated user manage their own profile, password, 2FA, and UI theme.

**Current Completion:** 85%

**Existing Features:**
- Profile edit (name/email), with email change resetting `email_verified_at`.
- Account deletion (self-service, password-confirmed via form request).
- Password change with current-password confirmation.
- 2FA setup/QR/recovery codes (Fortify's built-in flow), gated behind `password.confirm` middleware.
- Appearance (light/dark/system) toggle, persisted client-side.

**Missing Features:**
- No session list / "sign out other devices".
- No notification preferences.
- No account activity history visible to the user.

**Dependencies:** Laravel Fortify, `App\Http\Requests\Settings\*`.

**Database Tables:** `users`.

**Routes:** `profile.edit`, `profile.update`, `profile.destroy`, `security.edit`, `user-password.update`, `appearance.edit`.

**Permissions:** Any authenticated user (`auth` middleware; some routes additionally require `verified`).

## 3. Application Settings (Branding & SMTP)

**Purpose:** Let system administrators configure the app's branding (logos) and outbound mail (SMTP) without touching `.env`.

**Current Completion:** 85%

**Existing Features:**
- Login logo and app logo upload, stored on the `public` disk, with cleanup of the previous file on replace.
- SMTP host/port/username/password/encryption/from-address configuration, applied at runtime to the mailer (`AppSettingsService::applyToMailer()`), consumed by every OTP/credential email flow.
- Settings persisted as flexible key/value pairs (`AppSetting` model), not fixed columns.

**Missing Features:**
- No "send test email" button to verify SMTP works before relying on it.
- No validation/preview of logo dimensions or file size limits surfaced in the UI beyond backend rules.
- No versioning/audit of settings changes.

**Dependencies:** `App\Services\AppSettingsService`, `Illuminate\Support\Facades\Storage`, `Illuminate\Support\Facades\Mail` (config override).

**Database Tables:** `app_settings`.

**Routes:** `admin.settings.edit`, `admin.settings.update`.

**Permissions:** `systemAdminRoles` only.

## 4. Content Management (Support / About)

**Purpose:** Let system administrators edit the public-facing Support and About pages.

**Current Completion:** 80%

**Existing Features:**
- Two fixed content keys (`support`, `about`), each with a title + rich body, auto-created with placeholder content on first visit.

- Only two hardcoded content slots — no way to add arbitrary new content pages (FAQ, Privacy Policy, Terms, etc.) without a code change.
- No revision history / rollback.
- No rich-text sanitization visible at the model layer (relies on whatever the editor component does client-side) — worth a security pass since `body` is rendered as HTML.

**Dependencies:** `App\Models\AppContent`.

**Database Tables:** `app_contents`.

**Routes:** `admin.content.edit-support`, `admin.content.update-support`, `admin.content.edit-about`, `admin.content.update-about`, `support`, `about`.

**Permissions:** Editing restricted to `systemAdminRoles`; viewing is public (no auth).

## 9. School Self-Activation Onboarding

**Purpose:** Let a school claim its pre-provisioned record and set up its own login without admin intervention.

**Current Completion:** 90%

**Existing Features:**
- Public "find my school" lookup by School ID (rate-limited `throttle:10,1`).
- Activation form (school head, librarian, property custodian, contacts, email, location correction).
- Email-OTP verification (6-digit, 5-minute expiry, hashed in cache) before the account is actually created.
- Auto-generated user account + password, emailed to the school along with a one-time in-app credentials display screen.
- Graceful SMTP-unavailable fallback: activation request is still recorded and flagged for manual admin approval instead of hard-failing.
- Already-activated schools can still resubmit the form to update their own profile details (re-uses the same endpoint, branches on `is_activated`).

**Missing Features:**
- No explicit "resend OTP" action distinct from re-submitting the full form (re-submission works but re-sends the whole payload).
- No expiry warning/countdown UI feedback beyond the timestamp passed to the frontend (may exist client-side, not verified here since only backend was inspected).

**Dependencies:** `App\Services\SchoolActivationService`, `App\Services\AppSettingsService`, `App\Mail\SchoolActivationOtpMail`, `App\Mail\SchoolActivationCredentialsMail`.

**Database Tables:** `schools`, `users`, `cache` (OTP storage).

**Routes:** `school.find`, `school.activate.edit`, `school.activate.store`, `school.activate.verify-otp`, `school.activate.credentials`.

**Permissions:** Public/guest (no auth required to look up or activate a school; rate-limited instead).

## 11. Admin Executive Dashboard

**Purpose:** Division-wide, single-screen executive summary for superintendents, ASDS, and program managers.

**Current Completion:** 90%

**Existing Features:**
- KPI tiles: total/activated schools, learners, printed copies delivered + defect rate, equipment units + repair count, digital LMs + quality-assured count, pending deliveries.
- Four charts: enrollment by grade level (grouped columns), school activation by municipality (progress bars), equipment condition (diverging stacked bar), learning-resource defect rate by municipality (progress bars) — each with a table-view toggle for accessibility.
- Scope filters: District, School Level (`school_type`), Grade Level, plus an "Entire Division" reset — every aggregate query is consistently re-scoped through a single `$schoolIds` resolution.
- Pending school-activation-requests panel with quick links.
- Searchable/paginated school directory with inline edit/delete actions, carrying the active scope forward via hidden fields.

**Missing Features:**
- No date-range comparison (this year vs. last year) beyond the single active-school-year snapshot.
- No drill-down: clicking a chart segment doesn't navigate to the underlying filtered list.
- No saved/bookmarked filter views.
- No PDF/print export of the dashboard itself (CSV exports exist only on the separate Reports module).

**Dependencies:** Nearly every domain model (`Enrollment`, `LearningResource`, `IctEquipment`, `Sme`, `OtherEquipment`, `ResourceDistribution`, `DigitalLearningMaterial`, `School`), shared chart components (`ChartCard`, `GroupedColumnChart`, `HProgressBars`, `DivergingStackedBar`, `StatTile`).

**Database Tables:** Reads across `schools`, `enrollments`, `learning_resources`, `ict_equipment`, `sme`, `other_equipment`, `resource_distributions`, `digital_learning_materials`, `municipalities`, `districts`, `grade_levels`, `school_years`.

**Routes:** `admin.dashboard`.

**Permissions:** `adminWorkspaceRoles` (all 10 admin/staff roles can view; there is no separate "manage" action on this page besides the embedded directory edit/delete, which any workspace role can trigger since no additional role check gates those specific actions beyond the group-level middleware).

## 15. School Resource Encoding & Inventory Ledger

**Purpose:** Let each school record the printed learning resources it physically has on hand, and track day-to-day stock movement (issued, borrowed, damaged, lost, condemned).

**Current Completion:** 85%

**Existing Features:**
- Catalog-backed encoding (pick a division title, supply only quantities/delivery details) or fully manual entry (free-text title/type/publisher) when no catalog match exists.
- Bulk save endpoint: creates, updates, and deletes rows in one transaction based on a diff against what the school already has, with ownership verification (`abort_if` on foreign IDs).
- Six-state inventory ledger per resource (available/issued/borrowed/damaged/lost/condemned) auto-initialized on creation and auto-adjusted when encoded quantities are edited.
- Full movement history log (who, what type, quantity, from/to status, notes) per resource, capped at the latest 200 entries in the UI.
- Admin-side read-only monitoring across all schools (`AdminLearningMaterials`) and per-school detail view (`AdminSchoolShow`).

**Missing Features:**
- No low-stock or damage-rate threshold alerts/notifications.
- No barcode/QR label generation for physical book tagging.
- No transfer-between-schools workflow (a book can only be "received," not reassigned to another school without manual re-encoding).
- No pagination/date filtering on the movement history beyond a hard 200-row cap.

**Dependencies:** `App\Services\LearningResourceInventoryService`, `App\Models\ResourceTitle`.

**Database Tables:** `learning_resources`, `learning_resource_inventories`, `inventory_movements`.

**Routes:** `school.resources.index`, `school.resources.store`, `school.inventory.index`, `school.inventory.movements.store`, `admin.learning-materials.index`.

**Permissions:** Encoding/inventory actions are `role:school` only (own school); admin monitoring view is `coreMonitoringRoles`.

## 16. Resource Distribution (Division → School)

**Purpose:** Track the physical shipment of printed resources from the division office to individual schools, and the school's confirmation of receipt.

**Current Completion:** 80%

**Existing Features:**
- Admin creates a delivery against one school + one active catalog title + quantity, auto-generating a reference code (`DST-{year}-{id}`).
- Pending / Received / Cancelled lifecycle; cancel only allowed while pending.
- School-side "receive" action captures a damaged-quantity count and, on confirmation, automatically creates a new `LearningResource` record (with its inventory ledger initialized) so the delivery flows straight into the school's stock.
- Admin monitoring index with search, status filter, and per-status summary counts; school-side read-only list of its own deliveries.

**Missing Features:**
- One delivery = one school + one title at a time — no bulk/batch shipment creation (e.g., "ship Title X to 50 schools" in one action).
- Receiving the same title twice for the same school creates a **second, separate** `LearningResource` row rather than accumulating into the existing one — this fragments a school's on-hand count for a title across multiple ledger rows over time.
- No delivery-receipt (DR) document generation/printing.
- No partial-quantity receipt (a school can only report *damaged* quantity, not a partial *received* quantity vs. what was shipped).

**Dependencies:** `App\Services\ResourceDistributionService`, `App\Services\LearningResourceInventoryService`, `ResourceTitle`, `School`.

**Database Tables:** `resource_distributions`, `learning_resources`, `learning_resource_inventories`, `inventory_movements`.

**Routes:** `admin.distributions.index/store/cancel`, `school.distributions.index`, `school.distributions.receive`.

**Permissions:** Creation/cancellation restricted to `systemAdminRoles`; school receipt is `role:school` (own school only, ownership-checked).

## 18. ICT Equipment

**Purpose:** Track ICT hardware (laptops, desktops, tablets, phones, printers, projectors, smart TVs) from division catalog through per-school asset records.

**Current Completion:** 80%

**Existing Features:**
- Two-tier model: admin-maintained catalog (`IctEquipmentCatalogItem`) + per-school physical unit records (`IctEquipment`), each unit optionally linked back to its catalog entry.
- Full per-school CRUD (register/edit/soft-delete a unit) with rich asset metadata: serial/property number, barcode/QR code fields, acquisition date/cost/funding source, supplier, delivery info (IER no.), warranty expiry, useful life, current location, assigned personnel, condition (5 states), status (7 states).
- Automatic movement/audit log on every create/update/status-change/condition-change/reassignment/relocation/delete.
- Admin monitoring index across all schools with search/category/status filters and a status-count summary.
- Catalog CSV import with template; delete-guard when a catalog item is already in use (soft-deleted units still count, since the check uses `withTrashed()`).

**Missing Features:**
- `barcode`/`qr_code` are plain text fields only — no actual QR/barcode image generation, printing, or scan-to-lookup feature (matches the roadmap's "Pending: QR Labels").
- No warranty-expiry or maintenance-due notifications/alerts.
- No depreciation/asset-value reporting over time (`acquisition_cost` and `useful_life_years` are captured but not used in any calculation observed).
- No disposal/condemnation workflow beyond setting a `status` value manually (no approval chain, no disposal certificate).
- No inter-school transfer flow (equipment is tied to one school; moving it means delete + re-create elsewhere).

**Dependencies:** `App\Services\IctEquipmentService`, `App\Services\IctEquipmentCatalogImportService`.

**Database Tables:** `ict_equipment` (soft-deletes), `ict_equipment_catalog_items`, `ict_equipment_movements`.

**Routes:** `admin.ict-equipment.index`, `admin.ict-equipment-catalog.index/store/update/destroy`, `admin.ict-equipment-catalog.import.template/store`, `school.ict-equipment.index/store/update/destroy`.

**Permissions:** Admin monitoring view is `coreMonitoringRoles`; catalog management is `catalogRoles`; school-side CRUD is `role:school` (own school only).

## 20. Other Equipment/Materials

## 21. School Enrollment

**Purpose:** Let each school record male/female learner counts per grade level for the active school year, feeding every learner-dependent dashboard/report figure.

**Current Completion:** 85%

**Existing Features:**
- Simple per-grade-level male/female count form for the currently active school year.
- Save-all-at-once semantics: existing rows for the active year are replaced in a transaction (delete-then-recreate), so partial updates don't leave stale rows behind.
- Zero-count entries are silently dropped rather than stored as empty rows.
- Feeds directly into the Admin/School dashboards and the Reports adequacy calculation.

**Missing Features:**
- No section-level breakdown (only grade-level totals, no per-section rosters).
- No year-over-year comparison view within the Enrollment page itself (the data exists via `school_year_id`, but there's no UI to browse prior years side by side).
- No individual learner records (this is aggregate headcount only, by design, but worth noting for anyone expecting per-learner tracking).

**Dependencies:** `GradeLevel`, `SchoolYear`.

**Database Tables:** `enrollments`.

**Routes:** `school.enrollment.index`, `school.enrollment.store`.

**Permissions:** `role:school` only (own school).

## Cross-Cutting Observations

- **No Policy classes.** All authorization is route-level `role:` middleware plus manual `abort_if($resource->school_id !== $school->id, 403)` ownership checks scattered through controllers. This works but means authorization rules aren't centrally discoverable/testable the way Policies would be.
- **Three equipment verticals (ICT, SME, Other) are near-duplicate code** — same model shape, same controller shape, same service shape, same catalog+movement+CRUD pattern. This is consistent and well-tested, but a shared trait/base class could reduce triplication if a fourth vertical is ever added.
- **CSV is the only bulk I/O format** across every import/export feature in the app (schools, locations, resource titles, digital LMs, all three equipment catalogs, all four reports). No Excel/PDF import or export exists anywhere.
- **QR/barcode fields exist as plain text columns** on all three equipment tables but there is no image generation, printing, or scan-to-lookup capability — matches the product roadmap's "Pending: QR Labels" entry.
- **No notification/alerting system** anywhere (low stock, warranty expiry, pending activation aging, damaged-copy thresholds) — everything is pull-based (someone has to open a page to notice a problem).
- Test coverage is solid for the domains it covers (Dashboard, Distribution, Reports, Equipment ×3, Digital LMs, Location Management, Admin Settings, School Activation) — 207 passing feature tests observed at last run.
